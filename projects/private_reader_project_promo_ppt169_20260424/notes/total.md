# 01_封面

Private Reader 是一套面向团队与机构的私有阅读平台，我们希望它解决的不只是“看书”这一个动作，而是把私有书库、阅读体验、内容治理和后续平台扩展放在同一条产品线上。  
这一版展示会从项目定位、产品能力、后台链路、技术架构和阶段成果五个层面，说明它为什么值得继续投入。  
[停顿]

要点：① 私有阅读平台定位 ② 自托管、多用户、离线优先 ③ 具备继续演进价值  
时长：1分钟

# 02_项目定位

[过渡] 先看这个项目的定位，它并不是一个单点阅读工具，而是一套完整的平台型能力。  
它同时覆盖读者端体验、后台治理和资源入库链路，因此更适合被理解成一个可自托管的阅读基础设施。  
对于团队和机构来说，这意味着它可以承接私有书库、知识分发和阅读工作台等长期场景。  

要点：① 平台而非单功能 App ② 兼顾体验与治理 ③ 具备多端延展空间  
时长：1分钟

# 03_核心能力

[过渡] 在能力层面，第一阶段已经打通了四条最关键的业务链路。  
从登录与会话恢复，到书架进入阅读器，再到批注、书签和进度同步，以及后台管理和资源扫描，核心骨架已经搭起来了。  
这意味着项目现在不再停留在概念设计，而是已经具备可演示、可提交、可继续迭代的基础。  

要点：① 四条关键业务链路已打通 ② 三类角色与五个后台分区已可用 ③ 三类格式插件已纳入统一处理  
时长：1.5分钟

# 04_产品体验

[过渡] 接下来我们从真实界面看一下产品体验。  
当前 Flutter App 已经具备登录、书架、阅读器、设置和后台入口的连续路径，不是零散页面集合，而是可感知的完整体验。  
同时，手机与平板共用一套代码基线，阅读器也已经做了浮层化和主题化处理，具备进一步打磨的价值。  

要点：① 已形成完整用户路径 ② 统一 Flutter 基线覆盖手机和平板 ③ 阅读器交互与主题能力已具雏形  
时长：1.5分钟

# 05_管理与导入

[过渡] 如果说读者端代表使用价值，那么后台与导入链路代表平台价值。  
Private Reader 已经把用户、角色、书籍、批注和资源扫描纳入统一后台，并且支持 WebDAV 扫描源接入。  
这条链路最大的意义在于，它让内容不是靠人工逐本导入，而是可以通过扫描、解析和入库形成可重复流程。  

要点：① 后台管理能力完整 ② WebDAV 扫描链路已打通 ③ 平台具备内容治理与接入效率  
时长：1.5分钟

---

# 06_技术架构

[过渡] 再往下看，项目在技术架构上也不是临时拼装，而是有比较明确的边界设计。  
后端采用 Kotlin 加 Spring Boot，多模块拆分插件体系，前端则用 Flutter 作为统一基座，并为 Web 和 Desktop 的收敛预留路径。  
同时，AOT 友好和 Native-ready 的约束也说明，这个系统从一开始就考虑了交付效率和未来扩展。  

要点：① 后端、插件、前端分层清晰 ② 兼顾当前落地与未来跨端 ③ Native-ready 设计提升长期可交付性  
时长：1.5分钟

# 08_路线图与价值

[过渡] 最后一页想强调的是，Private Reader 的价值不止于已经完成了什么，更在于它接下来可以演进成什么。  
下一阶段会优先让后台能力落到 Flutter Web 和 Desktop，再逐步迁移书架、账户和阅读器能力，最终形成统一的多端阅读与管理平台。  
如果把时间线拉长来看，它沉淀的并不是一个单点产品，而是一套适合团队和机构长期使用的数字阅读基础设施。  

要点：① 后台先行推进多端化 ② 阅读能力将继续跨端收敛 ③ 项目具备长期平台化价值
