# Web端移除与Flutter多端路线

## 1. 当前决定

项目已经移除原 `web/` 目录下的 Next.js Web 端。

从当前节点开始：

- 原 Web 端不再保留代码目录
- 不再承接新功能开发
- 不再作为主交付目标
- 后续前端产品工作统一进入 `mobile/` Flutter 工程

## 2. 为什么停用原 Web 端

- 当前主交付能力已经集中到 `backend/ + mobile/`
- Flutter App 已经覆盖读者主链路与后台管理主链路
- 双前端并行演进会放大设计、联调、测试与维护成本
- 后续希望统一为 Flutter 多端，以共享阅读器、后台中心与业务状态管理

## 3. 原 Web 端移除策略

当前执行物理删除，原因如下：

- 避免误以为 Next.js Web 端仍可作为开发入口
- 避免继续维护第二套前端依赖、构建脚本和管理页面
- 让工程入口明确收敛到 `backend/ + mobile/`

但需要明确：

- 不再恢复 `web/` 目录作为主线前端
- 不再新增 Next.js 页面、组件或 Web 专用构建脚本
- 历史交互经验以文档和 Flutter 现有实现为准

## 4. 迁移目标

建立一套以 Flutter 为核心的统一前端：

- Mobile：Android，后续可继续补 iOS
- Web：通过 Flutter Web 承接浏览器访问与后台管理
- Desktop：Windows / macOS / Linux 桌面端

## 5. 建议的迁移顺序

### 阶段 A：基础收敛

- 继续稳定 `mobile/lib/app`
- 抽象通用导航壳与页面容器
- 收敛主题 token 与通用组件

### 阶段 B：后台先行

- 优先将后台管理能力适配到 Flutter Web/Desktop
- 原因是后台对键鼠、表格、列表、上传的需求清晰，且比阅读器更容易拆分

### 阶段 C：书架与账户

- 迁移登录、书架、个人中心
- 统一会话、配置与权限入口

### 阶段 D：阅读器跨端

- 处理 Web/Desktop 下的窗口化布局
- 增强键盘导航、鼠标选择、滚轮与快捷键
- 评估 EPUB/TXT/PDF 的跨平台渲染差异

## 6. 迁移时的架构原则

- 一个业务模型，多个平台适配层
- 尽量保留 `data` 与 `features` 层
- 平台差异优先收口在 `app`、少量 widget 和服务适配层
- 避免重新引入第二套前端状态体系

## 7. 对仓库的执行约束

- 新功能默认落在 `mobile/` 与 `backend/`
- 涉及浏览器和桌面的新能力，优先评估 Flutter 多端实现
- 不再添加 `web/` 目录、Next.js 依赖或 Web 专用构建脚本

## 8. 下一步建议

- 先完成 Flutter Web 的基础运行与路由壳层验证
- 把后台管理中心作为第一批 Web/Desktop 落地能力
- 将现有 Flutter 阅读器、后台中心和 API 客户端抽象为可复用的跨端能力
